[이전 글](../2025-07-19-onlyon-introduction)에선 OnlyOn 서비스를 제작하게 된 이유와 기획 단계를 정리해보았는데요. <br />
이번 글에선 기획 단계를 마친 후 기술 스택 선정 과정을 정리해보겠습니다.


### 1. 모노레포

먼저 모노레포 구조를 선택한 이유는 블로그와 포트폴리오 페이지를 독립적으로 운영하고 싶었습니다. <br/>
하나의 서버에서 운영되기 보다는 각 서비스는 의존없이 동작하는게 추후 기능이 확장될 때 유지보수가 쉽다고 생각했습니다.

예를들어 블로그의 포스팅을 업로드하였을 때 멀티레포 구조에서는 포트폴리오의 코드까지 다시 배포될 것입니다. <br/>
각 서비스에서 배포와 변경 사항 관련해서 영향을 끼치고 싶지 않아 모노레포 구조를 선택하게 되었습니다.

### 2. Next.js

저는 블로그에서 가장 중요한 것이 내 글이 필요한 사람들에게 전달되어 도움을 주는 것이라고 생각하는데요. <br/>
그러한 측면에서 사용자가 특정 키워드로 입력하였을 때 관련된 포스팅이 검색엔진에 노출되어야 하기 때문에 검색엔진 최적화가 중요하다고 생각하였습니다.

그래서 SPA 방식보다는 SSR 방식을 통해서 서버에서 페이지를 색인하여 검색엔진 크롤러가 내 페이지를 분석할 수 있도록 하기 위해 Next.js 프레임워크를 선택하게 되었습니다.

두번째 이유로는 현업에서는 React 18 환경에서 주로 개발하여서 서버 컴포넌트를 사용해보고 싶었고, <br/>
마침 블로그와 포트폴리오 페이지는 정적인 요소가 많이 때문에 서버 컴포넌트를 사용하여 페이지를 더 빠르게 렌더링 할 수 있을 것이라고 생각하였습니다.

### 3. Tailwind CSS

서버 컴포넌트와 연관지어 Tailwind CSS의 가장 큰 장점은 Zero-runtime 이라는 점이 있습니다. <br/>
런타임 환경에서 렌더링 되지 않고 빌드 시점에 모든 스타일이 적용되기 때문에 페이지 로딩 속도가 빠르다는 점이 있습니다.

또한 CSS in JS 방식 예를 들어 스타일 컴포넌트는 스타일과 컴포넌트가 분산되어있어 개발자 경험이 떨어진다고 생각하였습니다.

마크업을 개발하면서 하나의 컴포넌트 정의 파일에서 스타일까지 작성하는 것이 저와 더 맞아서 Tailwind CSS를 선택하게 되었습니다.

### 4. MDX

처음에는 블로그 글을 작성하기 위해 에디터를 직접 구현할까 생각하기도 하였습니다. <br/>
그런데 제가 이 서비스를 제작하는 목적이 블로그 글을 작성하는 것이 아니라 브랜딩 페이지를 만드는 것이고, <br/>
내가 직접 하나의 서비스를 운영하면서 디자인, 개발, 배포 이러한 프로세스를 경험하는게 우선이라고 생각하여 개발 리소스가 많이 발생하는 에디터 기능은 직접 구현하지 않고 외부 서비스를 사용하기로 결정하였습니다.


그래서 처음에 마크다운으로 글을 작성하려고 하였다가 gitbook을 사용하면서 느꼈던 단점이 글을 인터렉티브하게 <br/>
작성하기 어렵다는 점이 있던 것이 생각나서 처음 사용해보는 `MDX`를 선택하게 되었습니다.

MDX를 사용한 덕분에 이미지를 클릭하면 확대되는 기능도 추가할 수 있었고, 디자인 시스템을 정의하여 디자인 통일성을 유지할 수 있었습니다.

[Next.js 공식문서](https://nextjs.org/docs/pages/guides/mdx)와 프론트엔드 이현준 개발자님이 작성하신 [블로그 글](https://www.leehyeonjun.com/profile)을 통해 쉽게 연동할 수 있었습니다!
